# Generic RL Training Skill — Design Spec

## Goal

Make the rl-training skill project-agnostic. Today it's hardcoded for LeggySim (Leggy biped, lerobot SSH host, specific WandB project, specific metrics). After this change, the skill works for any RL project by reading project-local configuration and scripts generated by a setup agent.

## Architecture Overview

```
SKILL.md (generic orchestrator, lives in ~/.claude/skills/rl-training/)
    │
    ├── reads config.md        (.claude/rl-training/config.md — project-local, committed)
    ├── reads infra memory     (~/.claude/projects/<hash>/memory/rl_training_infra.md — gitignored)
    └── calls scripts          (.claude/rl-training/scripts/* — project-local, committed)
```

Phases: `SETUP → CLARIFY → CODE → LAUNCH → MONITOR/ITERATE`

## File Layout

### Skill (global)

```
~/.claude/skills/rl-training/
└── SKILL.md
```

Single file. No embedded scripts. No project-specific references. Pure orchestration logic that reads project-local config.

### Project-local (generated by setup agent)

```
.claude/rl-training/
├── config.md
└── scripts/
    ├── train.sh
    ├── kill_training.sh
    ├── monitor.py
    ├── evaluate_policy.py
    ├── init_session.sh
    └── get_latest_run.py
```

### Sensitive infra (gitignored, in project memory)

```
~/.claude/projects/<hash>/memory/
└── rl_training_infra.md
```

Contains SSH host/user, WandB project path, remote paths, API keys — anything that shouldn't be committed.

## Setup Agent

### Trigger Conditions

- **Full setup**: Skill triggered and no `.claude/rl-training/config.md` exists
- **Partial update**: User explicitly asks to update training config

### Step 1 — Codebase Exploration (autonomous)

No user interaction. The agent:
- Scans project structure, dependencies, entry points
- Identifies simulator, framework, algorithm, task names
- Looks at existing training scripts, configs, reward files
- Checks local hardware: GPU availability, OS
- Produces a findings summary for subsequent steps

### Step 2 — Robot & Objective (interactive)

- Present findings from Step 1, ask user to confirm/correct
- Ask about robot type, specificities, capabilities
- Ask about training objective
- One question at a time

### Step 3 — Training Infrastructure (interactive)

- Ask: local or remote training?
- If remote: ask for SSH details → stored in `rl_training_infra.md`
- If local: confirm GPU setup
- Ask about dependency management
- Confirm training command

### Step 4 — Monitoring & Evaluation (interactive)

- Ask what tools are available (WandB, TensorBoard, logs)
- If WandB: ask for project path → stored in `rl_training_infra.md`
- Ask about key metrics and what "good" vs "bad" looks like
- Ask about evaluation scenarios and metrics
- May iterate to nail down eval strategy

### Step 5 — Generate (autonomous)

- Write `config.md` from gathered info
- Write `rl_training_infra.md` to project memory
- Generate all 6 scripts adapted to the project
- Scripts use current Leggy scripts as structural reference (patterns, not literal content)
- Present summary of generated files to user for review

### Partial Updates

When config already exists:
- Show current config sections
- Let user specify which sections to update
- Skip to the relevant step, pre-fill with current values
- Only regenerate affected scripts

## Generic SKILL.md Structure

### Phase: SETUP

Gated on `config.md` existence. Runs the setup agent flow above. Exits to CLARIFY.

### Phase: CLARIFY

Same as today — ask user what they want to improve. Reads `config.md` for robot context and source files instead of hardcoded knowledge.

### Phase: CODE

Investigate → research → implement. Reads `config.md` → Source Files to know which files to modify. Uses robot specificities from config to make informed changes.

### Phase: LAUNCH

- Calls `.claude/rl-training/scripts/init_session.sh`
- Calls `.claude/rl-training/scripts/train.sh`
- Calls `.claude/rl-training/scripts/get_latest_run.py` to capture run ID
- Spawns Monitor Cron
- Reads infra memory for connection details if remote

### Phase: MONITOR (Cron)

- Calls `.claude/rl-training/scripts/monitor.py`
- Optionally calls `.claude/rl-training/scripts/evaluate_policy.py`
- Uses `config.md` → Monitoring section for thresholds and key metrics
- Discord notifications via the existing discord-notify skill (not embedded)
- Decision logic: keep monitoring / finish / kill+iterate

### Phase: ITERATE

Diagnose failure → fix code → relaunch. Uses config for source file paths and robot context.

### What Gets Removed from Current SKILL.md

- All Leggy-specific references (joint names, motor-to-knee, task name)
- Hardcoded script paths (e.g., `scripts/train_remote.sh`)
- Hardcoded WandB project
- Hardcoded remote host details
- Embedded Discord webhook logic
- Hardcoded metric names and categories

### What Stays Generic

- Multi-agent architecture (Run Agent + Monitor Cron)
- Phase flow and transitions
- Decision logic structure (consecutive bad → kill, max iterations → pause)
- File-based state machine (session_state.json)
- Instructions to read config.md and infra memory

## Script Generation Reference

The 6 scripts are generated per-project. The current Leggy scripts serve as structural reference for patterns, with all project-specific content (host names, task IDs, metric prefixes, joint names) parameterized from config.

| Script | Purpose | Key inputs from config |
|--------|---------|----------------------|
| train.sh | Launch training | Training.Command, Training.Execution, infra memory |
| kill_training.sh | Stop training | Training.Execution, infra memory |
| monitor.py | Fetch metrics → markdown | Monitoring.Tool, Monitoring.Metric categories |
| evaluate_policy.py | Headless eval | Evaluation.Scenarios, Evaluation.Metrics, Task.Name |
| init_session.sh | Session management | (generic, minimal config needed) |
| get_latest_run.py | Find active run | Monitoring.Tool, infra memory |

## LeggySim Migration

LeggySim is migrated directly — no interactive setup needed since everything is known:

1. Generate `.claude/rl-training/config.md` with current Leggy specifics
2. Generate `.claude/rl-training/scripts/` adapted from current `scripts/`, reading config instead of hardcoded values
3. Create `rl_training_infra.md` in project memory with SSH and WandB details
4. Existing `scripts/` directory stays as-is (committed, used by other things like `export_web.py`)
5. Generic SKILL.md replaces current Leggy-specific SKILL.md

## Out of Scope

- Discord webhook management (handled by discord-notify skill)
- Web export (project-specific, not part of generic training loop)
- Multi-robot / multi-task training in single project
